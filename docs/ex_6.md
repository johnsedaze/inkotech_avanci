## Тест-кейс 6: Уменьшение суммы РТУ-А при наличии фиксаций — предупреждения D009

### Контекст
Вариант тест-кейса 5. Дополнительно: сумма РТУ-А уменьшилась с 800 до 700
после того, как были зафиксированы распределения по 200 руб. в каждом этапе.
Фиксации (200) превышают новый плановый лимит РТУ-А в этапе (175) → D009.
Алгоритм генерирует предупреждения, фиксации не трогает, строки ожидания
сохраняются — численный результат совпадает с тест-кейсом 7.

---

### Входные данные

**Сумма заказа: 900 (уже уменьшенная)**

- Этап1 (25%): лимит = 225
- Этап2 (25%): лимит = 225
- Этап3 (50%): лимит = 450
- Пороги: 225 / 450 / 900

**Оплаты:**
- Оплата 000027: 300 (27.03.2026)
- Оплата 000028: 300 (29.03.2026)

**РТУ после уменьшения:**
- РТУ-А: **700** (28.03.2026) ← было 800
  - РТУ-А × Этап1 (25%): **175** ← было 200
  - РТУ-А × Этап2 (25%): **175** ← было 200
  - РТУ-А × Этап3 (50%): **350** ← было 400
- РТУ-Б: ожидается (не реализован), 100 руб.

**Фиксации (созданы когда РТУ-А = 800):**
- Оплата 000027 × РТУ-А × Этап1 = **200** | Аванс | ✓
- Оплата 000028 × РТУ-А × Этап2 = **200** | Зачёт оплаты | ✓

---

### Диагностика при пересчёте

**Проверка B (D009): фиксации по паре (РТУ × этап) > плановый лимит**

| Пара | Фиксации | Новый лимит | Превышение |
|---|---|---|---|
| РТУ-А × Этап1 | 200 | 175 | **+25** → D009 ⚠ |
| РТУ-А × Этап2 | 200 | 175 | **+25** → D009 ⚠ |

**Проверка A (D010): фиксации по этапу > лимит этапа**
- Этап1: фиксации=200 < лимит этапа=225 → OK
- Этап2: фиксации=200 < лимит этапа=225 → OK

*Предупреждения D009 — не критические. Алгоритм продолжает расчёт.*

---

### Анализ шага 1а

Цели оплат не изменяются (зависят только от сумм оплат и лимитов этапов заказа):
- Оплата 000027: НОВАЯ ОПЛАТА → целевой = Этап1
- Оплата 000028: ДОБИВКА, Idea 1 → целевой = Этап2 (обрезка от Этап3)

---

### Пошаговый расчёт Фазы 1

**Оплата 000027** (целевой = Этап1, ОстатокПлатежа = 300 − 200 = 100):

Построение списка доступных РТУ для Этап1:
- РТУ-А: ОстатокПоРТУОбщий = max(700 − 400 − 0, 0) = 300 > 0
  - ЛимитРТУВЭтапе = 700 × 25% = **175**
  - УжеРаспределено (фиксации Этап1 + авто) = 200 + 0 = 200
  - ОстатокРТУВЭтапе = max(175 − 200, 0) = **0** → не добавляем в список
- **База = 0** → строка ожидания = 25 (слот РТУ-А исчерпан фиксациями)

- ЕмкостьЭтапа1 = max(225 − 200 − 0, 0) = 25
- СуммаНаЭтап = min(100, 25) = 25 → строка ожидания 25 в Этап1
- Остаток = 100 − 25 = 75 → переизбыток

**Оплата 000028** (целевой = Этап2, ОстатокПлатежа = 300 − 200 = 100):

- Этап1: ёмкость = 225 − 200 − 25(авто) = 0 → пропуск
- Этап2: ЕмкостьЭтапа2 = max(225 − 200 − 0, 0) = 25
  - РТУ-А: ОстатокРТУВЭтапе = max(175 − 200, 0) = 0 → не добавляем
  - База = 0 → строка ожидания = 25
- Остаток = 100 − 25 = 75 → переизбыток

### Почему строки ожидания = 25, а не 0

В тест-кейсе 5 строки ожидания возникали потому, что:
- РТУ-А лимит в этапе = 200, использован полностью фиксациями → слот пуст

В тест-кейсе 6 причина другая, но эффект тот же:
- Новый РТУ-А лимит в этапе = **175**, но фиксации = **200** → слот **отрицательный**
- ОстатокРТУВЭтапе = max(175 − 200, 0) = 0 → РТУ-А исключается из распределения
- Строка ожидания = 25 (та же, что займёт РТУ-Б)

Разница отражается только в диагностике: D009 вместо "норма".

Уменьшение суммы РТУ при наличии фиксаций, превышающих новый лимит:
- Не ломает расчёт — алгоритм устойчив
- Генерирует предупреждения D009 (не ошибки)
- Фиксации остаются неизменными — это закрытый период, трогать нельзя
- Строки ожидания сохраняются в том же размере (25), ждут РТУ-Б
- Оставшаяся глобальная ёмкость РТУ-А (700 − 400 = 300) доступна для Этап3,
  если туда дойдут будущие оплаты
