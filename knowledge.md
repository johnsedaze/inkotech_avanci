# Knowledge Base

## Хэш входных данных для идемпотентного расчёта

### Общая практика

Хэш входных данных — устоявшаяся практика для предотвращения повторных вычислений и лишних записей в БД. Применяется под разными названиями:

- **ETL / Change Data Capture** — перед загрузкой строки вычисляют хэш всех полей и сравнивают с сохранённым. Если равен — строка не обновляется.
- **Кэширование вычислений** — в системах с дорогими пересчётами (биллинг, зарплата, себестоимость). Хэш входных параметров = ключ кэша результата.
- **Идемпотентные очереди** — в микросервисах (Kafka, RabbitMQ) хэш сообщения используют для дедупликации при повторной доставке.
- **1С (БСП)** — механизм "версионирования объектов": перед записью истории сравнивается хэш реквизитов объекта с предыдущей версией. Если не изменился — версия не создаётся.

**Требование к хэшу:** детерминированность — одни и те же входные данные должны всегда давать одинаковый результат. Для таблиц это обеспечивается фиксированной сортировкой (`УПОРЯДОЧИТЬ ПО`).

**Хэш-функция в 1С:** `ХешированиеДанных(ХешФункция.MD5)` → свойство `ХешСумма` возвращает тип `УникальныйИдентификатор` (128 бит = UUID). Хранить эффективнее чем строку: меньше места, нативное сравнение через `=`, хорошо индексируется.

Коллизии MD5 возможны только при целенаправленной атаке — для бизнес-данных достаточно.

### Применение к задаче: идемпотентный расчёт распределения оплат

#### Состав хэша входных данных

Три источника расчёта из функций-геттеров `РаспределитьОплатыПоЗаказу`:

**`ОплатыФакт`** (`ПолучитьОплатыПоЗаказу`): `Ссылка`, `Дата`, `СуммаПоЗаказу`  
**`Реализации`** (`ПолучитьРеализацииПоЗаказу`): `Ссылка`, `Дата`, `СуммаДокумента`  
**`ЭтапыПлан`** (`ПолучитьТаблицуЭтапов`): `УИДЭтапа`, `ПроцентЭтапа`, `ВариантОплаты`  
**`РеквизитыЗаказа`**: `СуммаДокумента` (сумма заказа)

**Что учитывается в хэше:**
- Сумма заказа
- Количество оплат + сумма и дата каждой
- Количество реализаций + сумма и дата каждой
- Количество этапов + процент и вариант оплаты каждого

**Что не нужно явно включать:**
- `СуммаЭтапа` — производная (`СуммаЗаказа × ПроцентЭтапа / 100`), покрывается двумя другими полями
- Проведённость документов — оба запроса фильтруют `И ... .Проведен = Истина`; если документ снять с проведения, он исчезнет из таблицы → изменится количество и суммы → хэш изменится автоматически

#### Функция формирования хэша

```1c
// Формирует MD5-хэш входных данных расчёта.
// Если хэш совпадает с предыдущим — расчёт можно пропустить.
// Детерминированность обеспечена: таблицы отсортированы в запросах.
//
Функция СформироватьХэшВходныхДанных(СуммаЗаказа, ОплатыФакт, Реализации, ЭтапыПлан)

    Части = Новый Массив;

    // 1. Сумма заказа
    Части.Добавить("З:" + Формат(СуммаЗаказа, "ЧДЦ=2; ЧРД=.; ЧГ="));

    // 2. Оплаты: количество + детали каждой
    //    Таблица отсортирована по (Дата, Номер, Ссылка) в ПолучитьОплатыПоЗаказу
    Части.Добавить("О:" + ОплатыФакт.Количество());
    Для Каждого Стр Из ОплатыФакт Цикл
        Части.Добавить(
            "о:" + Стр.Ссылка.УникальныйИдентификатор()
            + ":" + Формат(Стр.Дата, "ДФ=yyyyMMddHHmmss")
            + ":" + Формат(Стр.СуммаПоЗаказу, "ЧДЦ=2; ЧРД=.; ЧГ=")
        );
    КонецЦикла;

    // 3. Реализации: количество + детали каждой
    //    Таблица отсортирована по (Дата, Номер, Ссылка) в ПолучитьРеализацииПоЗаказу
    Части.Добавить("Р:" + Реализации.Количество());
    Для Каждого Стр Из Реализации Цикл
        Части.Добавить(
            "р:" + Стр.Ссылка.УникальныйИдентификатор()
            + ":" + Формат(Стр.Дата, "ДФ=yyyyMMddHHmmss")
            + ":" + Формат(Стр.СуммаДокумента, "ЧДЦ=2; ЧРД=.; ЧГ=")
        );
    КонецЦикла;

    // 4. Этапы: количество + УИД + процент + вариант оплаты
    //    Отсортированы по НомерСтроки в ПолучитьТаблицуЭтапов
    Части.Добавить("Э:" + ЭтапыПлан.Количество());
    Для Каждого Стр Из ЭтапыПлан Цикл
        Части.Добавить(
            "э:" + Стр.УИДЭтапа
            + ":" + Формат(Стр.ПроцентЭтапа, "ЧДЦ=2; ЧРД=.; ЧГ=")
            + ":" + Строка(Стр.ВариантОплаты)
        );
    КонецЦикла;

    СтрокаОтпечатка = СтрСоединить(Части, "|");

    // Сжимаем до УникальногоИдентификатора (MD5 = 128 бит = UUID)
    Хешер = Новый ХешированиеДанных(ХешФункция.MD5);
    Хешер.Добавить(СтрокаОтпечатка);
    Возврат Хешер.ХешСумма; // тип УникальныйИдентификатор

КонецФункции
```

#### Точка применения в `РаспределитьОплатыПоЗаказу`

```1c
РеквизитыЗаказа = ОбщегоНазначения.ЗначенияРеквизитовОбъекта(ЗаказСсылка, "СуммаДокумента, Договор");
СуммаЗаказа = РеквизитыЗаказа.СуммаДокумента;

Если СуммаЗаказа = 0 Тогда
    ИНК_ДиагностикаРаспределенияОплат.ЗавершитьДиагностику(КонтекстДиаг);
    Возврат;
КонецЕсли;

ЭтапыПлан  = ПолучитьТаблицуЭтапов(РеквизитыЗаказа.Договор, СуммаЗаказа);
ОплатыФакт = ПолучитьОплатыПоЗаказу(ЗаказСсылка);
Реализации = ПолучитьРеализацииПоЗаказу(ЗаказСсылка);

// --- Проверка идемпотентности ---
ХэшВхода = СформироватьХэшВходныхДанных(СуммаЗаказа, ОплатыФакт, Реализации, ЭтапыПлан);
Если ХэшВхода = ПолучитьСохранённыйХэш(ЗаказСсылка) Тогда
    // Входные данные не изменились — пропускаем расчёт полностью
    ИНК_ДиагностикаРаспределенияОплат.ЗавершитьДиагностику(КонтекстДиаг);
    Возврат;
КонецЕсли;

// ... весь расчёт ...

Набор.Загрузить(ТаблицаИтог);
Набор.Записать(Истина); // запись в регистр распределений

СохранитьХэш(ЗаказСсылка, ХэшВхода); // ВНУТРИ той же транзакции
```

> **Важно:** `СохранитьХэш` выполняется в той же транзакции что и `Набор.Записать`. При откате транзакции хэш не сохранится — данные останутся согласованными.

#### Хранение хэша

Отдельный регистр сведений `ИНК_СостояниеРасчётаОплат`:

| Тип поля | Имя | Тип данных |
|----------|-----|------------|
| Измерение | `ЗаказКлиента` | `ДокументСсылка.ЗаказКлиента` |
| Ресурс | `ХэшВходныхДанных` | `УникальныйИдентификатор` |

#### Эффект

```
Регламент обрабатывает ЗаказА (1-й раз)
  → хэш изменился → расчёт → запись в регистр → сохранение хэша

Регламент снова обрабатывает ЗаказА (дубль из-за гонки)
  → хэш тот же → Возврат; (0 обращений к регистру распределений)
```
