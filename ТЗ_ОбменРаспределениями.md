**Версия:** 1.1  
**Дата:** 06.04.2026  

**Связанные материалы:**

- `Идея ТЗ по обмену.md` — исходные требования

## 1. Цель и границы задачи

### 1.1. Цель

Обеспечить перенос зафиксированных строк распределения оплат из регистра сведений ERP `ИНК_РаспределениеОплатПоЭтапам` в учётные документы базы бухгалтерии (БП) через **HTTP-сервис 1С**, с обратной фиксацией результата в ERP (признак `ВыгруженоВБП`).

### 1.2. Участники

| Роль | Конфигурация | Роль в обмене |
|------|----------------|---------------|
| Источник данных | ERP: Управление предприятием | Отбор строк, вызов сервиса БП, установка `ВыгруженоВБП` |
| Приёмник | БП (типовая бухгалтерия, см. `metadata_bp.md`) | Поиск документов по публичным идентификаторам, запись связей «оплата ↔ реализация», ответ ERP |

### 1.3. Вне скоупа (если не согласовано отдельно)

- Первичная синхронизация документов ERP→БП и заполнение регистра `ПубличныеИдентификаторыСинхронизируемыхОбъектов` (предполагается существующим обменом).
- Изменение алгоритма распределения в ERP (`ТЗ_РаспределениеОплат_v7.md`).
- Выгрузка строк без документа **Реализация** (см. п. 4.3).

## 2. Термины

- **Строка распределения** — одна запись регистра сведений `ИНК_РаспределениеОплатПоЭтапам` с полным набором измерений (ключ записи).
- **УИД документа ERP** — `УникальныйИдентификатор()` ссылки на документ в ERP, строкой (36 символов), в том виде, в каком он хранится в `ПубличныеИдентификаторыСинхронизируемыхОбъектов.Идентификатор` в БП.
- **ПДС (ERP)** — документ `ПоступлениеБезналичныхДенежныхСредств`.
- **Поступление на р/с (БП)** — документ `ПоступлениеНаРасчетныйСчет` (аналог оплаты безналом в БП по составу обмена).
- **РТУ (ERP)** — `РеализацияТоваровУслуг`.
- **Реализация (БП)** — `РеализацияТоваровУслуг`.
- **База БП (целевая)** — информационная база бухгалтерии, в которую отправляется HTTP-запрос; набор таких баз задаётся регистром `ИНК_ПараметрыПодключенийКБазамДанных`.
- **Ключ маршрутизации** — ИНН организации заказа клиента, по которому выбирается строка регистра параметров подключения (и далее URL HTTP).

## 3. Источник данных в ERP

### 3.1. Регистр сведений `ИНК_РаспределениеОплатПоЭтапам`

Структура по `метаданные.md`:

**Измерения:**

- `ЗаказКлиента` — Документ.ЗаказКлиента  
- `Реализация` — Документ.РеализацияТоваровУслуг  
- `ДокументОплаты` — Документ.ПоступлениеБезналичныхДенежныхСредств  
- `УИДЭтапа` — Строка  
- `Зафиксировано` — Булево  

**Ресурсы:**

- `Сумма` — Число  
- `ТипОперации` — Перечисление.ИНК_ТипыОпераций (`Аванс`, `ЗачетОплаты`)  
- `ЭтоПереизбыток` — Булево  

**Реквизиты:**

- `ВыгруженоВБП` — Булево  

### 3.2. Условие отбора строк к выгрузке

В пакет обмена попадают записи, для которых одновременно:

1. `Зафиксировано = Истина`  
2. `ВыгруженоВБП = Ложь`  
3. **Дополнительно (обязательное уточнение для реализации):** `Реализация` заполнена (не пустая ссылка).  
   - *Обоснование:* в БП по описанию задачи связь оформляется между **поступлением на р/с** и **реализацией**; строки «ожидания реализации» без `Реализация` в ERP не имеют пары документов в БП в смысле п. 4.2 идеи.

При необходимости выгрузки иных случаев (например, только аванс без РТУ) — отдельное согласование и доработка ТЗ.

### 3.3. Порядок обработки

- Рекомендуемый порядок сортировки внутри **одной целевой базы БП**: `(ДокументОплаты.Дата, ДокументОплаты.Номер, ДокументОплаты)`, затем `(Реализация.Дата, Реализация.Номер)`, затем `УИДЭтапа` — для предсказуемости повторных запусков.
- При нескольких базах сначала группировка по **ИНН организации** (см. п. 3.4), внутри группы — сортировка как выше.

### 3.4. Определение базы БП: регистр `ИНК_ПараметрыПодключенийКБазамДанных` и ИНН организации

Параметры подключения к базам бухгалтерии **не задаются константой**: они хранятся в регистре сведений **`ИНК_ПараметрыПодключенийКБазамДанных`** (`метаданные.md`).

**Структура регистра (используемые поля):**

| Элемент | Назначение |
|---------|------------|
| Измерение `ИмяБазы` | Логическое имя записи (идентификатор строки настройки). |
| Ресурс `ИНН` | ИНН организации, которой соответствует данная база БП. |
| Ресурс `ПодключениеHTTP` | Строка подключения к **HTTP**-сервису БП (базовый URL или шаблон — по фактическому заполнению внедрения). |
| Ресурс `СерверHTTP` | Дополнение к адресу HTTP при необходимости (хост/префикс — по фактическому заполнению). |
| Ресурсы `СтрокаПодключения`, `Сервер`, `БазаДанных`, `Пользователь` | Используются при **интеграциях не по HTTP** (COM и т.п.); для обмена по HTTP-сервису опорными считаются **`ПодключениеHTTP` / `СерверHTTP`**, если иное не зафиксировано в проекте. |

**Правило выбора базы для строки распределения:**

1. По измерению **`ЗаказКлиента`** строки регистра `ИНК_РаспределениеОплатПоЭтапам` получить **`Организацию`** заказа (реквизит документа `ЗаказКлиента` в ERP).  
2. По ссылке на **`Организации`** получить **`ИНН`** организации (без обращения «через точку» в цикле — пакетно, например `ОбщегоНазначения.ЗначениеРеквизитаОбъектов` / запрос).  
3. В регистре `ИНК_ПараметрыПодключенийКБазамДанных` найти запись, у которой ресурс **`ИНН`** совпадает с ИНН организации заказа (сравнение после нормализации: убрать пробелы, единый формат 10/12 знаков).  
4. Из найденной записи взять параметры HTTP и сформировать итоговый URL вызова метода обмена (конкатенация/правило — в коде ERP, единообразно для всех баз).

**Если для ИНН записи в регистре нет или не заполнен HTTP-адрес:** строки распределения с такой организацией **не отправлять** в БП; зафиксировать ошибку маршрутизации (`CONFIG_ERROR` / аналог), **`ВыгруженоВБП` не менять**. Поведение при **нескольких** строках регистра с одним и тем же `ИНН` — **открытый вопрос 10** (запрет дублей на уровне данных или приоритет по `ИмяБазы`).

### 3.5. Несколько организаций и несколько баз БП: стратегия соединений

В одной выборке к выгрузке строки могут относиться к **разным организациям**, значит — к **разным URL** баз БП. Требуется явно выбрать и закрепить в реализации одну из стратегий (либо комбинацию).

**Вариант A — группировка без «прогрева» (рекомендуется по умолчанию для HTTP):**

1. После отбора строк по п. 3.2 дополнить каждую строку вычисленным **`ИНН`** и ссылкой на параметры HTTP (или ключом `ИмяБазы`).  
2. **Сгруппировать** строки по **`ИНН`** (или по однозначному ключу целевой базы).  
3. Для каждой группы сформировать **отдельный HTTP-запрос** (пакет JSON только по строкам этой базы) на URL, полученный из регистра.  
4. Объединить ответы и применить п. 6 к каждой строке по её `registryKey`.

*Плюсы:* нет лишних вызовов «в пустую», проще таймауты и диагностика по базе. *Особенность:* HTTP **без состояния** — отдельного «соединения» как у COM-подключения нет; «держать открытым» нечего, кроме **кэша параметров** (соответствие ИНН → URL, заголовки аутентификации) в памяти на время выполнения задания.

**Вариант B — кэш параметров + опциональный прогрев доступности:**

1. В начале выполнения регламентного задания (или перед первой отправкой в рамках ИНН) по **уникальным ИНН** из текущей выборки загрузить из регистра параметры и положить в **соответствие** `ИНН → параметры HTTP` (и при необходимости предразобранный URL).  
2. Опционально для каждого уникального URL выполнить **лёгкий запрос** (например, существующий `PingMe` сервиса `ИНК_Бухгалтерия`) для раннего обнаружения недоступности базы.  
3. Далее — как вариант A: группировка и пакетные POST по базам.

*Плюсы:* меньше повторных чтений регистра при большом числе строк на один ИНН; ранний сигнал о «мёртвой» базе.

**Вариант C — «прогрев всех баз из регистра»:**

1. При старте задания прочитать **все** записи `ИНК_ПараметрыПодключенийКБазамДанных`, для каждой валидной строки с заполненным HTTP выполнить опциональный Ping (или только заполнить кэш).  
2. Обрабатывать выборку распределений с использованием этого кэша.

*Плюсы:* актуальность проверки всех баз независимо от текущей выборки. *Минусы:* лишняя нагрузка и время, если в выборке одна организация; при большом числе записей в регистре — избыточно.

**Рекомендация ТЗ:** реализовать **вариант A** как основной; кэш из **варианта B** — обязательно внутри одного запуска задания для снижения обращений к регистру; **вариант C** — только по флагу настройки («проверять все базы при старте») при требовании эксплуатации.

**Сортировка и отбор «по организации»:** отбор строк к выгрузке по п. 3.2 остаётся прежним; **фильтрация по одной организации** допускается только в **ручном** сценарии (п. 7.2) для целенаправленной выгрузки в одну БП. Регламентное задание обрабатывает **все** отобранные строки, маршрутизируя их по группам ИНН.

## 4. Действия в БП

### 4.1. Поиск документов по УИД

Использовать регистр сведений **`ПубличныеИдентификаторыСинхронизируемыхОбъектов`** (`http/metadata_bp.md`):

- Измерение `Идентификатор` — строка (УИД из ERP).  
- Измерение `Ссылка` — ссылка на объект БП.  
- Измерение `УзелИнформационнойБазы` — узел плана обмена.  
- Реквизит `ТипМетаданных` — строка.  

**Для каждой строки распределения:**

1. По УИД документа оплаты ERP найти ссылку на **`ПоступлениеНаРасчетныйСчет`** в БП.  
2. По УИД документа реализации ERP найти ссылку на **`РеализацияТоваровУслуг`** в БП.  

**Открытый вопрос 1:** по какому именно **`УзелИнформационнойБазы`** (какому плану обмена) отбирать записи регистра, если для одного УИД теоретически возможны несколько строк? В ТЗ закрепить единственный узел (константа/параметр сервиса) или правило выбора (например, приоритет по типу обмена ERP↔БП).

**Открытый вопрос 2:** нужно ли дополнительно фильтровать по **`ТипМетаданных`** и каково ожидаемое значение для оплаты и реализации (строка из обмена).

### 4.2. Соответствие типа операции ERP и табличных частей БП

По смыслу задачи (из идеи ТЗ):

| `ТипОперации` (ERP) | Документ, который изменяется в БП | Смысл связи |
|---------------------|-----------------------------------|-------------|
| `Аванс` | `ПоступлениеНаРасчетныйСчет` | В расшифровке платежа указывается **документ расчётов** = реализация (погашение/привязка к сделке — по правилам типовой БП для выбранного способа погашения). |
| `ЗачетОплаты` | `РеализацияТоваровУслуг` | В табличной части **`ЗачетАвансов`** добавляется (или актуализируется) строка: **`ДокументАванса`** = поступление на р/с, **`СуммаЗачета`** = сумма из строки распределения. |

**Ссылка на метаданные БП:**

- `ПоступлениеНаРасчетныйСчет` — табличная часть **`РасшифровкаПлатежа`**: реквизиты в т.ч. `Сделка` (документ расчётов), `СуммаПлатежа`, `СпособПогашенияЗадолженности`, `ДоговорКонтрагента` и др.  
- `РеализацияТоваровУслуг` — табличная часть **`ЗачетАвансов`**: `ДокументАванса`, `СуммаЗачета` (см. `metadata_bp.md`).

**Открытый вопрос 3:** для варианта `Аванс` подтвердить штатный сценарий типовой БП:

- заполняется ли **`РасшифровкаПлатежа.Сделка`** ссылкой на **`РеализацияТоваровУслуг`** и какой **`СпособПогошенияЗадолженности`** должен использоваться;  
- или аванс в БП отражается иначе (отдельная операция/вид оплаты), и требуется ссылка на процедуру из типовой конфигурации.

### 4.3. Правила записи документов

1. Оба документа в БП должны быть **найдены**. Иначе строка **не обрабатывается**, в ответе — код/текст причины, в ERP `ВыгруженоВБП` **не** устанавливается.  
2. Документы не должны быть **помечены на удаление** (иначе — отказ с диагностикой).  
3. **Блокировка от изменений** (закрытый период, запрет редактирования, блокировка сеансом и т.п.): запись **не выполняется**, возвращается информативное сообщение, в ERP `ВыгруженоВБП` остаётся `Ложь`.  
4. После изменения табличной части документы подлежат **проведению** (если были проведены) в порядке, исключающем нарушение учётных связей (уточнить: сначала Поступление, затем Реализация или наоборот — **открытый вопрос 4** в соответствии с типовой логикой зачёта).  
5. **Идемпотентность:** повторная отправка той же строки (тот же состав ключа измерений и та же сумма) не должна приводить к задвоению сумм в БП. Варианты реализации (выбрать при разработке однозначно):  
   - поиск существующей строки ТЧ с тем же «партнёром» связи (ПДС↔РТУ) и той же суммой;  
   - или хранение в ERP/БП служебного ключа строки обмена (расширение — только по согласованию).  

**Открытый вопрос 5:** допускается ли **корректировка суммы** уже выгруженной строки (повтор с другой суммой) — если да, описать правило перезаписи в БП и сброс `ВыгруженоВБП` в ERP.

### 4.4. Валюта и сумма

- В первой версии обмена передаётся **`Сумма`** из регистра ERP **как есть** (валюта взаиморасчётов считается согласованной между парами документов обмена).  
- **Открытый вопрос 6:** если валюты/курсы в паре документов БП расходятся с ERP — отказать с диагностикой или пересчитывать по курсу документа БП.

## 5. Транспорт: HTTP-сервис

### 5.1. Размещение кода

- **БП:** расширить HTTP-сервис `ИНК_Бухгалтерия` (`RootURL`: `bp`, см. `ИНК_Бухгалтерия.xml`) новым шаблоном URL и обработчиком; модуль — по аналогии с `http/http_module.bsl` (кодировка UTF-8, формирование JSON-ответа).  
- **ERP:** общий модуль / обработка регламентного задания:  
  - определение целевого URL по п. **3.4** (регистр **`ИНК_ПараметрыПодключенийКБазамДанных`**, сопоставление по **ИНН организации** документа **`ЗаказКлиента`**);  
  - при нескольких организациях — **несколько вызовов** HTTP на разные базы согласно п. **3.5** (пакет запроса не смешивает строки разных ИНН);  
  - фактический способ сборки URL из полей **`ПодключениеHTTP`** и **`СерверHTTP`** зафиксировать в коде ERP единообразно для всех баз (при необходимости — отдельная служебная функция `СобратьАдресHTTPБП` с параметром записи регистра параметров).

### 5.2. Рекомендуемый режим обмена: пакетный

**Сравнение (п. 5 идеи ТЗ):**

| Подход | Плюсы | Минусы |
|--------|--------|--------|
| Вызов по одной строке | Проще отладка, меньше таймаутов на один запрос | Больше HTTP-накладных расходов, больше round-trip |
| **Пакет (рекомендуется)** | Один round-trip, один сеанс на стороне БП, проще единая транзакционная политика | Больше тело запроса/ответа; нужны лимиты размера пакета и таймаут |

**Рекомендация ТЗ:** один метод **POST** (или GET с телом не используется — предпочтительно **POST**) принимает **массив строк** и возвращает **массив результатов** по каждой строке. Опционально — параметр `?batchSize` на стороне ERP при нарезке больших выборок **внутри одной базы БП** (после группировки по ИНН, п. 3.5).

### 5.3. Формат запроса (проект)

Кодировка: UTF-8, `Content-Type: application/json`.

Массив элементов; каждый элемент — объект:

| Поле | Тип | Обязательность | Описание |
|------|-----|----------------|----------|
| `paymentUid` | Строка | Да | УИД `ДокументОплаты` (ERP) |
| `realizationUid` | Строка | Да | УИД `Реализация` (ERP) |
| `stageUid` | Строка | Да | `УИДЭтапа` (для трассировки и логов) |
| `fixedFlag` | Булево | Да | Всегда `true` (контроль целостности) |
| `operationType` | Строка | Да | `Аванс` или `ЗачетОплаты` (имя значения перечисления ERP) |
| `amount` | Число | Да | `Сумма` |
| `isExcess` | Булево | Нет | `ЭтоПереизбыток` — для фильтрации/логирования |
| `registryKey` | Объект | Да* | Состав ключа записи регистра ERP для однозначной идемпотентности и ответа (*структура полей — см. п. 5.5) |

**Объект `registryKey` (проект):**

- `order` — УИД заказа клиента (строка)  
- `payment` — УИД оплаты  
- `realization` — УИД реализации  
- `stageUid` — строка  
- `fixed` — булево (`true`)  

Так ERP сможет сопоставить ответ с набором записей регистра без двусмысленности.

### 5.4. Формат ответа (проект)

HTTP 200 при успешной **обработке запроса как целого** (парсинг прошёл), даже если часть строк завершилась ошибкой.

Тело JSON — объект:

- `success` — Булево: `Истина`, если все строки обработаны успешно.  
- `processedAt` — Дата/строка ISO — время на сервере БП.  
- `items` — массив результатов **в том же порядке**, что и в запросе (обязательно закрепить в реализации).

Элемент `items[]`:

| Поле | Тип | Описание |
|------|-----|----------|
| `status` | Строка | `OK` \| `NOT_FOUND` \| `LOCKED` \| `VALIDATION_ERROR` \| `WRITE_ERROR` \| `DUPLICATE` \| `CONFIG_ERROR` (на стороне ERP при отсутствии маршрута по ИНН — не приходит из БП, а проставляется до вызова) |
| `message` | Строка | Текст для журнала/администратора |
| `registryKey` | Объект | Эхо ключа из запроса |

При критической ошибке сервиса (500) ERP **не** меняет `ВыгруженоВБП`; предусмотреть повтор регламентного задания.

### 5.5. Безопасность

- Аутентификация вызова HTTP (basic, token в заголовке, IP whitelist) — **открытый вопрос 7**, зафиксировать в настройках ERP/веб-сервера.  
- Выполнение в привилегированном режиме на стороне БП только в границах обработчика и только при необходимости (как в текущем `SalaryStatement`); не расширять права без аудита.

## 6. Действия в ERP после ответа БП

Для **каждой** строки запроса, для которой в `items[]` пришёл `status = OK`:

1. Выполнить запись в регистр `ИНК_РаспределениеОплатПоЭтапам` с установкой реквизита **`ВыгруженоВБП = Истина`** для записи с соответствующим ключом измерений (без изменения `Зафиксировано` и ресурсов распределения).

Для строк с любым иным статусом:

- **`ВыгруженоВБП` не менять** (`Ложь`).  
- Зафиксировать результат в техническом журнале / регистре диагностики (при наличии — `ИНК_ДиагностикаРаспределенияОплат` или отдельный журнал обмена — **открытый вопрос 8**).

**Транзакция ERP:** целесообразно обновлять флаги **после** успешного разбора ответа; пакетное обновление набором записей по ключам строк со статусом `OK`.

## 7. Регламентное задание и ручной запуск

### 7.1. Регламентное задание (ERP)

- Периодичность: согласовать (например, каждые 15–60 минут в рабочее время).  
- Действия: отбор по п. 3.2 → получение ИНН по `ЗаказКлиента` (пакетно) → **группировка строк по целевой базе БП** (п. 3.4–3.5) → для каждой группы: нарезка пакетов (лимит N строк) → вызов HTTP на URL этой базы → обработка ответа по п. 6.  
- При ошибке сети/таймаута по конкретной базе: для затронутых строк флаги не менять; остальные группы (другие ИНН) обработать в том же запуске, если это реализовано без общей откатывающей транзакции на весь сеанс.  
- Кэш «ИНН → URL/заголовки» в пределах одного запуска задания — по п. 3.5 (вариант B).

### 7.2. Ручной запуск

- Команда/обработка в ERP: «Выгрузить распределения в БП» с отбором (по заказу, по периоду оплат, по конкретным строкам — **открытый вопрос 9**).  
- Использовать тот же программный интерфейс, что и регламентное задание.

---

## 8. Нефункциональные требования

- **Производительность:** целевой размер пакета по умолчанию 50–200 строк **на одну базу БП** (настраиваемо); при превышении времени ответа — уменьшать пакет. Учитывать, что N групп по ИНН даёт до N последовательных (или ограниченно параллельных — не входит в ТЗ) циклов вызова.  
- **Журналирование:** фиксировать длительность вызова **по каждому URL/ИНН**, количество строк в пакете, число `OK` / ошибок, обезличенный идентификатор пакета (GUID сеанса обмена), при необходимости — `ИмяБазы` из регистра параметров.  
- **Повторный запуск:** безопасен за счёт идемпотентности (п. 4.3 п. 5).

## 9. Тестирование (приёмочные сценарии)

1. Оба документа в БП есть, документы не заблокированы, `ТипОперации = ЗачетОплаты` → в РТУ появляется/обновляется зачёт, ERP `ВыгруженоВБП = Истина`.  
2. То же для `Аванс` → корректное заполнение расшифровки Поступления (после уточнения п. 4.2).  
3. Отсутствует оплата в `ПубличныеИдентификаторы...` → `NOT_FOUND`, флаг в ERP не меняется.  
4. Отсутствует реализация в регистре → `NOT_FOUND`.  
5. Документ заблокирован → `LOCKED`.  
6. Повторная выгрузка той же строки → не удваивает суммы в БП, `OK` или `DUPLICATE` с явной семантикой.  
7. Пакет из 3 строк: 1-я `OK`, 2-я ошибка, 3-я `OK` → в ERP обновляются только 1 и 3.  
8. Две строки распределения с заказами **разных организаций** (разные ИНН) → два HTTP-запроса на разные URL из регистра `ИНК_ПараметрыПодключенийКБазамДанных`, флаги обновляются независимо по результатам каждого ответа.  
9. ИНН организации заказа **отсутствует** в регистре параметров → строка не вызывает БП, диагностика маршрутизации, `ВыгруженоВБП` не меняется.
